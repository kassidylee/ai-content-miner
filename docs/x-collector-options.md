# X 关键词采集现状与候选方案

更新时间：2026-07-22

## 当前结论

本分支中的 Twikit 接口已经完成命令编排、配置检查、数据标准化、时间窗口、去重、失败
退出和单元测试，但**尚未通过真实 X 请求验证，当前不能用于每日推送**。

在 Python 3.11 和 `twikit==2.3.3` 环境中执行本地会话创建时，真实登录返回：

```text
Exception: Couldn't get KEY_BYTE indices
```

失败后没有生成 Cookie 文件，密码也没有被项目保存。该错误发生在 Twikit 解析 X 网页并
生成请求交易标识的阶段，早于账号凭证校验，因此更换显示名称、用户名格式或重复输入
密码无法解决。

Twikit 已有完全相同且仍开放的
[上游问题 #409](https://github.com/d60/twikit/issues/409)。从本地安装包代码和问题状态
判断，X 网页 JavaScript 已不再匹配 Twikit 2.3.3 用来提取 `KEY_BYTE` 索引的规则。

本分支的测试使用模拟客户端验证项目自身逻辑，不访问 X。测试通过只能说明桥接器的
配置、分页、转换、去重和失败处理符合预期，不能证明非公开 X 接口仍然兼容。

## 候选方案

| 方案 | 适配程度 | 主要优点 | 主要限制和风险 | 当前建议 |
| --- | --- | --- | --- | --- |
| X 官方 Recent Search API | 很高 | 官方接口；覆盖最近 7 天，符合每日推送窗口；维护风险最低 | 需要开发者账号、Bearer Token 和按量付费额度 | 稳定性优先时首选 |
| `twscrape` | 较高 | 支持关键词搜索、异步 API、JSONL CLI、本地 SQLite 会话；推荐直接导入浏览器 `auth_token` 和 `ct0` | 仍依赖非公开 GraphQL；Cookie 可能失效，账号可能受限；必须用真实账号低频验证 | 当前最值得做小规模技术验证的开源候选 |
| `Scweet` | 较高 | 支持关键词和日期搜索、同步/异步调用、JSON 输出；可从网页资源刷新查询标识 | 同样依赖非公开 GraphQL；本地模式仍需浏览器 Cookie；功能和状态管理范围大于本项目需要 | 作为第二开源候选 |
| 等待 Twikit 修复 | 无需改造 | 保留当前桥接代码和数据映射 | 修复时间未知；当前真实登录被阻塞 | 只适合作为观察项 |

### 1. X 官方 Recent Search API

X 官方文档说明，Recent Search 可读取最近 7 天内匹配查询的帖子：

- [Search recent Posts](https://docs.x.com/x-api/posts/search-recent-posts)
- [Usage and Billing](https://docs.x.com/x-api/fundamentals/post-cap)

项目按天运行，七天窗口已经满足产品需求。该方案需要新增 Bearer Token 配置、分页和官方
响应到项目文章格式的映射，并需要在开发者控制台购买和限制用量。费用应在真实关键词和
每日条数确定后通过小规模运行测量，不能仅凭静态文档估算。

### 2. twscrape

[`vladkens/twscrape`](https://github.com/vladkens/twscrape) 提供异步关键词搜索、账号会话
池和 JSONL CLI。其文档将包含 `auth_token` 与 `ct0` 的浏览器 Cookie 作为更稳定的账号
建立方式，这可以避开当前 Twikit 的用户名/密码登录步骤。

调研时上游最新发布为 `v0.19.2`（2026-07-21）。如果选择它，下一步只做一个专用账号、
一个关键词、少量结果的只读烟雾测试，并继续将 Cookie 和 SQLite 会话文件放在 Git 忽略
目录。验证内容至少包括：真实搜索、最近七天过滤、字段完整性、退出码、Cookie 失效和
限流行为。

### 3. Scweet

[`Altimis/Scweet`](https://github.com/Altimis/Scweet) 支持关键词、日期范围、同步/异步
调用及 JSON 输出。其文档明确不支持用户名/密码自动登录，而是读取浏览器 `auth_token`；
也提供从 X 网页资源刷新 GraphQL 查询标识的选项。

调研时最新发布为 `v5.0`（2026-03-20）。它仍然依赖非公开接口，项目自身关于“当前可用”
的陈述必须通过本地真实烟雾测试验证，不能直接当作兼容保证。

## 暂不采用的做法

- 不在本项目中复制或修改 Twikit 的交易标识算法。该算法紧跟 X 网页实现，维护成本高且
  很容易再次失效。
- 不把 Cookie、密码、Token、账号池数据库或代理凭证提交到 GitHub。
- 不为绕过限制加入 CAPTCHA 处理、自动注册、代理轮换或多账号批量登录。
- 不在选定替代方案前同时安装多个 X 爬虫依赖，以免扩大依赖和审查范围。

## 建议的下一步

1. 保持本 PR 为 Draft，不合并 Twikit 实验接口。
2. 团队先决定优先级：官方接口的稳定性，或非官方接口的零 API 费用。
3. 如果继续开源非官方路线，优先用 `twscrape` 做最小只读烟雾测试；通过后再替换桥接器，
   而不是同时保留两个默认 X 采集器。
4. 无论选择哪条路线，都复用当前标准 JSONL、最近七天窗口、去重和非零失败退出约定。
5. 本次调研和后续采集器替换均不调整内容筛选、评分体系或企业微信消息格式。
